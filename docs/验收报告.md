# 追溯验收四类场景报告

## 场景 1：容器栈启动后批量注入完整数据

执行：
```bash
docker compose up -d --build
docker compose exec backend python -m app.self_check
docker compose logs simulator
```

预期：
- 启动 PostgreSQL、Redis、Backend、Frontend 四个容器
- 自检脚本创建 1 个合格件、1 个不合格件
- 模拟器额外灌入 28 个构件的全流程数据
- 数据库存在多项目、多工厂、多工地记录

## 场景 2：构件追溯全链路

入口：登录任意账号 → 「构件追溯」。

输入追溯码 `PC-DEMO-PASS`（自检脚本生成）：

- 应展示「构件生产 → 构件出厂 → 运输轨迹 → 构件进场 → 构件吊装 → 节点连接 → 隐蔽验收 → 成品保护 → 档案归档」共 8 步以上
- 各步骤的 actor（参与方）、时间、关键信息完整

接口验证：
```bash
TOKEN=...  # 任意账号 token
curl -H "Authorization: Bearer $TOKEN" http://localhost:8000/api/trace/PC-DEMO-PASS
```

## 场景 3：进场验收不合格的构件在吊装环节无法被调用

构造不合格件：自检脚本会创建 `PC-DEMO-FAIL`，进场验收结论 = 不合格。

调用：
```bash
curl -X POST http://localhost:8000/api/site/hoisting \
  -H "Authorization: Bearer $CONTRACTOR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"component_id": <FAIL_ID>, "hoisted_at": "...", "equipment_no": "TC9999", "signal_worker": "X", "rigger": "Y", "coord_lng": 0, "coord_lat": 0, "result": "一次就位"}'
```

预期：HTTP 409，detail 包含「禁止进入吊装流程」。

## 场景 4：档案归档电子档案包按城建档案规范导出与报送

入口：登录 owner01 → 「档案归档」。

操作：
1. 选择已「成品保护」的构件 → 「生成档案」→ 状态变为「已生成」
2. 点击「下载 ZIP」→ 浏览器下载 `ARCH-*.zip`，解压包含：
   - `manifest.json`（档案号、规范版本、参与方权责矩阵）
   - `trace_timeline.json`（全链路时间线）
   - `cover.txt`（封面）
3. 点击「报送」→ 状态变为「已报送」（mock 端点默认接受）

接口验证：
```bash
curl -X POST http://localhost:8000/api/archives/generate/<component_id> \
  -H "Authorization: Bearer $OWNER_TOKEN"

curl -X POST http://localhost:8000/api/archives/<archive_id>/submit \
  -H "Authorization: Bearer $OWNER_TOKEN"
```
